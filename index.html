SYSTEM VS SOFTWARE

SYSTEM - The entire thing. Start to end. All components and softwares working in harmony.

SOFTWARE - Part of a system. A single unit of functional responsibility.

SYSTEMS -> SUBSYSTEMS/COMPONENTS -> MODULES -> FUNCTIONALITIES

System = "System is an organized structured entity, that consists of various elements that interact with each other to perform specific tasks or functions"

Software = "A software is a set of programs that are dependent on each other. There is sharing of resources between these programs and they are connected with each other such that there is less dependency on one another"

- Interactions must occur between components through an interface (API)

Types of Systems:

  • Data processing System - any system that involves lot of data (collection, manipulation, storage, retrieval)
  • Real Time System - any system that must act and provide full functionality within a given time constraint => hard - airbags, soft - digital signal processing
  • Decision Support System - represent data in the form of a model and then fit incoming input to the model to get output to make decisions (What if analysis, Data visualisation)
  • Knowledge Base System - The system uses a knowledge base which is a combination of laws, rules (which may expand) which relies on predicate logic to take the incoming input and using logical connectives, it forms a coherent output decision. (Explanation systems, knowledge acquisition)

REQUIREMENTS ENGINEERING/(DEFINITION)

  1. Identify Stakeholders
  2. Feasibility Study
    • Is it computationally possible?
    • Is your team and company equipped to do it?

    Feasibility Report

    • Technology (can tech support)
    • Budget
    • Time constraint
    • Skills of team
  3. Requirement Elicitation and Analysis
    • get the requirements
    • understand the requirements which are ambiguous
    • understand the customer's problems
    • Get problem statement
    • Define problem scope - list of all things that system must take care of (objectives and constraints)

    Elicitation

    • Interview Customer
      • formal/informal
      • open/closed ended questions
      • questionnaires
    • Brainstorm use cases
      • record the meetings
    • Ethnography
      • observe/recorded info
      • gather info
    • Prototyping
      • throw away prototyping

    Analysis

    • Traceability
    • Scope definition
    • Prioritisation

    * Generate use cases and user stories -> Generate Use Case Diagram

  4. Requirement Specification
    • specify the entire requirements into very specific targets and goals - functional + non functional reqs.
    • Create SRS report
    • SRS = Requirements + Systems Models
    • Create system models and diagrams
      • ER diagram
      • Use case diagram
      • State transition diagram
      • Data flow diagram
  5. Requirements Validation
    • check if the requirements are what the client wanted
    • Verify in team
    • Validate with client
  6. Managing Changes

Types of Requirements:

User/Abstract Requirements - Generic description without details.

System Requirements - exact detailed requirements

Extras

  • Platform Reqs. - specific to the dev platform being used (eg- login with clerkjs)
  • Process Reqs. - specific constraints to choose process model (waterfall, agile)
  • Organisational Reqs.
    • Environmental
    • Operational
    • Development
  • External Reqs. - ethical, legal, regulations
  • Enduring Reqs. - stable, never ending reqs. derived from core activity
  • Volatile Reqs. - reqs. likely to change a lot
  • Mutable Reqs. - reqs. changing due to system dev env
  • Emergent Reqs. - something that emerges with better understanding
  • Consequential Reqs. - reqs. that are a consequence of the system itself
  • Compatible Reqs. - reqs. that enforce compatibility of system with other systems

Functional Requirements : The main functional task that the app should perform

  • Inputs (format)
  • Outputs (format)
  • Computations
  • Data being stored
  • Timing and synchronisation

Non Functional Requirements : Auxiliary requirement that is not specific to the use case/functionality of the system but is useful for quality and performance of the system

  • Security - CVSS Score
  • Portability - % of target - number of systems it works on / number of target systems
  • UI - contrast score, general ui rules
  • Ease of Use - number of clicks, training time, number of help frames, usability testing score, number of screens navigated in time
  • Quick response time - latency in ms, throughput
  • Short processing latency - latency in ms
  • Availability - total uptime / total time
  • Reliability - Mean time between failure, rate of failure
  • Robustness - Mean time to recovery, probability of data corruption on failure, % of events causing failure
  • Resource Efficiency - % graph of resource use (resource utilization graph)
  • Maintainability - Code complexity, mean time to repair

Domain Requirements : Legal and ethical req.

  • Legal, ethical, regulation, statutory
  • Domain expert, legal team, insurance team

PROCESS MODELS

  1. WATERFALL MODEL
    • Linear development model
    • reqs. are clear and complete
    • go step by step. do not go back after step complete

    Communication --> Planning --> Modelling --> Construction --> Deployment

    Problems:

    • cannot accommodate any changes. Very rigid and inflexible
    • difficult to have all reqs. at the beginning
    • no deliverable till very late in the project life span
    • linear nature causes blocking states among teams
    • not suitable for fast paced projects with changing needs
  2. V MODEL
    • lot of testing is needed during the creation
    • waterfall + testing

    Requirement Engg. ===== Acceptance Testing

    Architecture Design ==== System Testing

    Component Design ==== Integration testing

    Code Generation === Unit Testing

  3. INCREMENTAL MODEL
    • Linear process + parallel implementation
    • well defined reqs.
    • create core functionality first and then work on improvements as increments
    • used when a basic deliverable is needed first (core product needed quickly)
    • priority modules first
    • every increment gives us some deliverable
    • each increment has time duration, budget
    • used when limited team (reduced blocking states) and large project scope with simple core need
  4. RAD - Rapid Application Development
    • for extremely tight deadlines (2-3 months)
    • highly skilled team, large team (many teams), parallel working
    • communication and planning done normally
    • modelling and construction done in rapid parallel teams (like incremental)
    • incremental model emphasizing short lifecycle
    • not good for complex projects with high risk
  5. EVOLUTIONARY MODEL
    • requirements are not set and are ever changing
    • linear method unrealistic

    Two types

    • Exploratory
      • create and keep validating with client
      • make changes to the code without discarding it
      • ends up with poor code quality due to many changes
      • make part by part of project with client validation
    • Throwaway Prototyping
      • a simplified implementation
      • compromised and low quality
      • validate with client and throw it away
      • start from scratch each time
      • high quality end product

    Prototype : Quick comm, quick design, compromised design and implementation, quick evaluation

    Cons:

    • Invisible process
    • poorly structured system

    Pros:

    • changing reqs.
    • better understanding reflected
  6. SPIRAL MODEL
    • Risk driven process model
    • follows evolutionary model in iterative approach
    • planning includes risk analysis in every cycle/iteration
    • made for perfecting the model than understanding the reqs.
    • large scale systems and softwares. critical software.
    • risk management and analysis given importance
    • ATC system, Payments, NASA etc.

AGILE PROCESS MODEL

  • collaborative, adaptive approach
  • modern process model
  • discussion oriented with team and customer
  • highly skilled and motivated team

4 Pillars

  • Iterative Dev - based on regular feedback
  • Collaboration - cross functional teams with shared goal
  • Flexibility - easy to adapt new reqs.
  • Continuous Improvement - team management and process refinement

TERMS OF AGILE

  • Product Backlog
    • List of all tasks to be completed for the product
    • user stories, features, bug fixes, implementation of functionalities, enhancements etc
    • complete list of tasks
    • sorted in priority
    • dynamic list
  • Sprint Backlog
    • a subset of the product backlog
    • items to be completed within the given sprint
    • 1-2 weeks to achieve sprint goal
  • Burn Down Chart
    • list of remaining tasks shown in graphical format
    • y axis shows remaining work and x is time
    • tracks day by day progress
    • goes down over time
  • Burn up Chart
    • list of completed tasks in graphical format
    • ulta of burn down chart
    • goes up over time
  • User Stories
    • A simplistic explanation of the expected user experience to provide context to the developer
    • An explanation of the use case

  • UI/UX Team

    • to ensure quality assurance by testing

  • Business Analyst
